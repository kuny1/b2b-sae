# b2b-sae
B2B 行业 Java EE 架构前端演进：B2B System Architecture Evolution

- 方案产出：依赖 ClaudeCode + DeepSeek API + 多轮对话


## 背景设定
公司的 B2B 平台（Java EE + jQuery）已上线运行超过 5 年，功能模块超过 30 个，涵盖供应商管理、招投标、合同、订单、物流、对账结算、审批流等完整链路。近两年业务部门的需求非但未减少，反而因组织调整、合规要求、客户定制等因素持续涌入。但交付速度却从“两周一个迭代”降至“两个月难以交付一个小变更”，另外还收到了多例反馈的问题：

- “改一个字段校验，要搜 12 个文件”
> 物料主数据加一个“危险品证书号”字段，前端需要改新增页、编辑页、详情页、批量导入页、列表导出模板、移动端审批页，外加至少三个 JSP 内联脚本。需求 BUG 数居高不下
- “用户说页面显示已通过，后台却是待审，不知道信哪个”
> 状态散落在 DOM 属性、隐藏域、多个 JS 全局变量中。不同开发者在不同时期写的代码各自维护一份状态副本。运气好才能不出问题
- “审批人点驳回没反应，排查半天发现是某个隐藏 div 里有必填校验”
> 交互逻辑深度依赖 DOM 的可见性和脚本加载顺序，操作结果不可复现，偶发问题只能靠重启解决
- “采购员和主管同时操作同一个订单，保存时互相覆盖，数据丢了要财务补录半天”
> 没有前端版本冲突检测，操作者只能看到一句“保存失败，请刷新重试”，实际已造成数据不一致
- “新人入职三个月才能独立改一个页面，还担心改出别的 bug”
> 模块间通过全局类名和事件委托隐式耦合，没有清晰边界，修改影响面无法静态评估，全靠老员工脑图和经验判断
